# Dropbox / Google Drive System Design — Fast Revision Notes

## Core Goal
Design a large-scale file storage and sync system like Dropbox / Google Drive that supports:
- Large file uploads/downloads
- Multi-device sync
- Resumable uploads
- High data integrity
- Low latency transfers

---

# Functional Concepts

## Blob Storage
- Store raw file bytes in blob/object storage (S3 / GCS equivalent)
- Optimized for large objects
- Cheap and horizontally scalable
- Never store raw file bytes in application DB

## Metadata Database
Stores structured data only:
- file_id
- owner_id
- filename
- mime_type
- size
- timestamps (created_at, updated_at)
- blob_storage_link
- chunk_list
- upload_status
- compression_algo (optional)
- folder_id

---

# Upload Path Concepts

## Pre-Signed URLs
Temporary secure upload/download links generated by server.

### Why
- Client uploads directly to blob storage
- Avoids server bandwidth bottleneck
- Avoids API gateway body size limits
- Removes double upload hop

### Flow
1. Client sends file metadata to server
2. Server creates metadata record
3. Server requests pre-signed URL from blob storage
4. Server returns URL to client
5. Client uploads directly to storage

---

## Chunking (Multipart Upload)

Split large files into small chunks (ex: 5 MB each).

### Benefits
- Supports very large files (10–50+ GB)
- Enables parallel uploads
- Enables resumable uploads
- Enables delta sync later
- Reduces retry cost

---

## Resumable Uploads

Track chunk status in metadata DB.

Each chunk stores:
- chunk_id
- fingerprint/hash
- upload_status
- chunk_storage_link

Resume logic:
- Fetch metadata
- Compare chunk list
- Upload only missing chunks

---

## Fingerprinting (Hashing)

Compute hash of chunk/file bytes (SHA-256 etc).

### Uses
- Unique chunk identity
- Integrity validation
- Deduplication
- Delta sync comparison

Content-based identity is safer than index-based identity.

---

## Upload Validation — Trust But Verify

Problem:
Client says “chunk uploaded successfully” — cannot blindly trust.

### Option A — Verify on Server
- Client reports upload success
- Server checks blob storage
- Then marks chunk complete

### Option B — Storage Event Notifications
- Blob storage emits upload event
- Server updates metadata automatically

Tradeoff:
- Server verify = simpler
- Storage events = more robust, more infra

# Sync Concepts

## Client Sync Agent
Installed app handles sync logic.

Contains:
- Local sync folder
- Local metadata DB
- Sync engine

---

## Local File Change Detection

Use OS file watchers:

- Windows → FileSystemWatcher
- macOS → FSEvents
- Linux → inotify

Triggers upload when local file changes.

---

## Remote Change Detection

Client must detect remote updates.

Two main approaches:

### Polling (Preferred)
Client periodically checks:

“Give changes since timestamp T”

Pros:
- Simple
- Reliable
- Cheap

Enhancement:
Adaptive polling frequency based on activity.

---

### Event Stream + Cursor (Advanced)

Use event log (Kafka-style).

- Every file change emits event
- Client keeps cursor
- Replay events after cursor

Pros:
- Version history
- Audit trail
- Recovery support

Cons:
- Higher complexity
- Not required unless versioning needed

---

# Delta Sync

Only transfer changed chunks instead of full file.

Requires:
- Chunking
- Chunk fingerprints
- Chunk timestamps

Flow:
- Detect changed chunks
- Download/upload only those chunks
- Client reassembles file

Major bandwidth saver.

# Data Integrity Concepts

## Multi-Layer Integrity Checks

- Chunk hashes
- File hash
- Upload verification
- Metadata vs storage cross-check
- Periodic reconciliation

---

## Reconciliation Job

Periodic full consistency check.

Steps:
1. Fetch remote metadata
2. Compare with local DB
3. Compare fingerprints
4. Repair mismatches

Used to fix drift from failures or bugs.

---

# Performance Optimizations

## Parallel Chunk Upload
Upload multiple chunks simultaneously to maximize bandwidth.

---

## Adaptive Chunk Size
Adjust chunk size based on:
- Network speed
- Failure rate
- Latency

---

## Compression

Compress before upload when useful.

Good candidates:
- Text
- Documents
- Logs

Avoid compressing:
- JPEG
- PNG
- MP4
- Already compressed media

Store compression algorithm in metadata.

Tradeoff:
CPU time vs network savings.

---

## CDN (Optional)

Content Delivery Network speeds downloads via edge nodes.

Useful when:
- Files are widely shared
- Global access patterns

Less useful when:
- Users mostly download their own files
- Data already region-local

Always justify CDN — don’t add blindly.

---

# Latency Improvements Summary

- Direct-to-storage upload (pre-signed URLs)
- Chunk parallelism
- Delta sync
- Selective compression
- Regional storage
- Optional CDN

---

# Consistency Model

Prefer:
Availability > Strong Consistency

Accept:
Eventual consistency for sync delays.

Users tolerate seconds of lag, not service outages.
